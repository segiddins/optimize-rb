# Spec: Harness (`load_iseq` override)

**Part of:** [Ruby the Hard Way](2026-04-19-talk-structure-design.md)
**Related:** [Optimizer core](2026-04-19-optimizer.md)

## Purpose

The harness is the single integration point between plain Ruby and the
optimizer. It intercepts every iseq Ruby loads, runs the optimizer
pipeline on it, and hands the rewritten iseq back to the VM. End users
do not call the optimizer directly; they just `require` their code.

## Shape

An override of `RubyVM::InstructionSequence.load_iseq(path)` that:

1. Delegates to the built-in compiler to get the original iseq
2. Checks the opt-out magic comment
3. Parses inline `@rbs` comments in the source into a type environment
4. Runs the optimizer pipeline
5. Returns the rewritten iseq (or the original, on opt-out or bail)

There is prior exploration in `experiments/02-load-iseq-hook/` — that
experiment confirmed the hook mechanism works; this spec defines the
production-ish shape.

## Scope policy

**Everything is optimized by default.** Files opt *out* with a magic
comment at the top:

```ruby
# rbs-optimize: false
```

This matches the talk's "love letter to a bad idea" framing: the
default is maximum ambition. Turning it off is the escape hatch, not
the on-ramp.

## Timing

**Eager.** Optimization runs synchronously inside `load_iseq`, before
the file's top-level code executes. No lazy first-call rewriting, no
runtime profiling. Two consequences:

- Load time goes up. This is a talk, not a product; that's fine.
- We only have static information at optimize time — no runtime type
  feedback. RBS and the contract are all we get, which is the point.

## RBS source

**Inline `@rbs` comments only.** Parsed from the source file via an
`rbs-inline`-flavored reader. No sidecar `.rbs` files for this project.

Rationale: inline comments show up next to the code in slides, which
is what the talk needs. Sidecar files would be invisible to the
audience.

## Failure behavior

If the optimizer pipeline raises or declines the whole file, the
harness returns the original iseq unchanged. A warning goes to stderr
(or the log sink — see optimizer spec) so the user knows a file was
not optimized and why.

Per-iseq decisions ("skipped this call site because…") are handled by
the pass itself and flow into the shared log.

## Not in scope

- Caching optimized iseqs to disk (`.load_from_binary` style)
- Reloading / `autoload` interactions beyond what `load_iseq` already
  provides
- Any runtime-profile-guided optimization
- Optimizing iseqs that were not loaded from a source file (e.g.
  `eval`, `instance_eval` with a string). Those just pass through.

## Open questions

- Exact parser choice for inline `@rbs` (rbs-inline, rbs gem, hand-roll)
- Whether the magic comment also supports enabling/disabling specific
  passes (`# rbs-optimize: inline,const-fold`) or only whole-file
  on/off. Default assumption: whole-file on/off only, for simplicity.
